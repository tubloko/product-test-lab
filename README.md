# ProductTestLab

A strategy workspace for media buyers: turn raw product ideas into structured test hypotheses before any money goes into ads.

## What it is

Most failed paid tests don't fail because the product is bad. They fail because the team didn't decide WHO buys it, WHY they buy it, and HOW the page reflects that — and started spending anyway. ProductTestLab is the layer between "I saw a product on a spy tool" and "the campaign is live": a workspace that forces a clean answer to those three questions before media spend starts.

The product is opinionated. It doesn't give you a blank canvas like Notion. It gives you rails — viability gating, avatar/angle/landing trinity, expected-vs-actual KPI tracking, versioned hypotheses. The friction is the feature.

It sits next to its sibling, AdTestLab, which handles the post-launch side (diagnosing what's actually happening after ads run). Together they form the full loop: strategy → spend → diagnosis → next-version strategy.

## Status

Work in progress. Built bottom-up, currently at the end of E7 of the planned roadmap. What works today:

- Email/password and Google authentication, single-user (no teams in MVP)
- Pipeline dashboard with real-time product list grouped by status (idea, researching, brief ready, testing, tested, killed, scaling) and per-card status menus
- Minimal Quick Add (single product by name, immediate save)
- Full 9-step Guided Wizard: product basics → niche & price → viability check (AI-summarized scorecard with risk flags) → multi-avatar selection (1–3 from 5 AI-generated) → per-avatar angle pick → AI-prefilled landing match → offer → expected KPIs (with AI baseline suggestion) → review → Save & Finish, which creates the product and N hypotheses atomically
- Resume / Discard interrupted wizard sessions from a dashboard banner
- Trinity enforcement (avatar pain + angle hook + landing hero) at the Zod schema layer, not just UI — a hypothesis cannot be marked "ready to test" with an incomplete trinity
- AI service layer with six Anthropic-backed endpoints, global monthly budget cap, per-call usage logging

What does NOT yet exist:

- Bulk Quick Add (spreadsheet-style table, paste, OG-scrape) — planned
- Hypothesis editor (entering actual results, writing learnings, creating v2 from learnings) — planned next
- Hypothesis tree visualization and side-by-side compare
- Settings page (display name edit, AI usage stats, account deletion)
- AdTestLab integration (auto-pull of actual results)

The full product spec lives in `ProductTestLab_Specification.md`.

## How it works

There are two ways to add a product, both valid for the same user at different moments:

**Capture mode** is for when ten product ideas just landed in front of you and you want to save them before they evaporate. Today this is the minimal "+ Quick add" — a single name field, saved to the Idea column. The full spreadsheet/paste/CSV/URL-scrape version is on the roadmap.

**Discovery mode** is the Guided Wizard. It's a 9-card flow that takes about ten minutes and produces 1–3 fully-formed test hypotheses linked to a product. Each card auto-saves; you can leave mid-flow and resume from the dashboard the next day with all state intact.

Inside the wizard, the heart of the product is the **viability check** (card 3) and the **trinity build** (cards 4–6):

- *Viability* asks eight questions about market saturation, competition, sourcing, brand risk, creative availability, wow factor, and affordable CPA. An AI summary turns those answers into a scorecard with risk flags and a recommendation. The user can save the product as Killed right there if the recommendation is "don't test" — saving money on a bad product is the single highest-value action this app does.
- *Trinity* is avatar (WHO) + angle (WHY) + landing match (HOW). The AI generates candidates, the user picks, the page-side message is forced to continue the ad-side promise rather than drift to features. A hypothesis without a complete trinity is allowed to live as a draft but cannot be promoted to "ready to test".

Once a hypothesis is tested in the wild (whatever ad platform), actual KPIs and learnings will be entered through the Hypothesis Editor (next epic), and a v2 can be generated by AI based on what v1 learned. This compounding loop is the point of the product — every test should make the next one smarter.

## Quick start

Requirements:

- Node.js ≥ 20.9 (the repo ships an `.nvmrc` at 20.9.0)
- A Firebase project with Firestore + Email/Password and Google providers enabled
- An Anthropic API key

```
git clone <repo> product-test-lab && cd product-test-lab
nvm use
npm install
cp .env.example .env.local
# Fill .env.local with your Firebase web config, Firebase admin SDK credentials, and ANTHROPIC_API_KEY
firebase deploy --only firestore:rules
npm run dev
```

Open http://localhost:3000, sign up, click `+ Guided setup`, walk through the wizard, click `Save & finish` on card 9.

Scripts:

- `npm run dev` — local dev server
- `npm run build` — production build
- `npm run lint` — ESLint check
- `npm run type-check` — `tsc --noEmit`
- `npm run test` — Vitest (currently exercises `lib/*`; no component tests yet)

## Stack

Framework: Next.js 16 (App Router, React Server Components, route groups for auth boundaries) with React 19 and TypeScript in strict mode.

Styling: Tailwind CSS v4 with CSS-first configuration (no `tailwind.config.js`; tokens live in `app/globals.css` under `@theme inline`). shadcn primitives built on `@base-ui/react` (with a few from `@radix-ui` where base-ui has no equivalent). Class merging via `clsx` + `tailwind-merge`. Toasts via `sonner`. Animations via `framer-motion` for the wizard slide transitions and staggered AI result reveals. Icons from `lucide-react`.

Data: Firebase Web SDK (Firestore + Auth) on the client; Firebase Admin SDK on the server for ID-token verification, budget tracking, and AI usage logging. Real-time subscriptions through `react-firebase-hooks/firestore`. No external state library — domain state flows through small custom hooks per collection (`useProducts`, `useHypotheses`, `useWizardSession`, etc.).

Forms: `react-hook-form` v7 with `@hookform/resolvers/zod`. Schemas (`zod` v4) live in `types/*.ts` and are reused by forms and API route validators — same schema, two consumers.

AI: Anthropic SDK calling Claude Sonnet 4.5, non-streaming, JSON-via-prompt structured output, parsed and Zod-validated server-side. Six endpoints under `app/api/ai/*` for avatar generation, angle generation, landing match suggestion, KPI baselines, viability summary, and v2 suggestions from learnings.

Testing: Vitest with `@vitest/coverage-v8`. Currently covers `lib/*` only.

## Architecture

The repo is organized by *kind* (component, hook, service, type), not by feature, mirroring the AdTestLab convention.

```
app/                # Next.js App Router
  (app)/            # authenticated route group
    layout.tsx      # auth guard + minimal shell
    page.tsx        # pipeline dashboard
    wizard/[sessionId]/page.tsx
  (auth)/login/page.tsx
  api/ai/<endpoint>/route.ts  # 6 AI endpoints
  layout.tsx        # root + theme bootstrap + Toaster
  globals.css       # tailwind + theme tokens

components/
  ui/               # shadcn primitives (button, input, card, dialog, select, tabs, …)
  wizard/           # wizard-specific cards + helpers
  forms/            # form dialogs (QuickAddDialog)
  icons/            # custom SVG components
  *.tsx             # other shared components (ProductCard, Pipeline, EmptyState, ResumeWizardBanner, StatusMenu)

hooks/              # one custom hook per file, useXyz pattern
  use{User,Login,AuthBootstrap,StartWizard,…}.ts
  use{Products,Product,ProductMutations,…}.ts
  use{Ai*}.ts       # 6 client wrappers for the AI endpoints

lib/
  firebase/         # Firebase init, paths helper, converters, per-collection services
  claude/           # Anthropic client, prompts, schemas, parse, client-side fetch helper
  utils.ts          # cn() class merge helper

types/              # zod schemas + inferred types, one file per domain entity
  index.ts          # the only barrel in the project — re-exports the rest
```

The data model under `users/{uid}/` has `products/{productId}` with nested `hypotheses/{hypothesisId}` and singleton `viability/current`, plus a sibling `wizardSessions/{sessionId}` and an `aiUsage/{logId}` analytics collection. Security rules enforce strict per-user ownership and deny client access to admin-only namespaces (`system/*`, `feedback/*`).

## Key design decisions

**AI as the default path, not a sidebar.** Every creative task — avatars, angles, landing copy, viability synthesis — leads with AI. Manual entry is always available but never the primary flow. The product assumes 2026 user expectations.

**Opinionated rails over flexibility.** Trinity is enforced at the Zod layer. Viability is the first card of the wizard, not a footnote. A hypothesis cannot enter "ready_to_test" status with an empty hero message. Inflexibility around the right things is the moat — Notion is the alternative and Notion is not opinionated.

**No fake intelligence.** The AI does not score products on a 1–10 scale, does not predict winners, does not pretend to know the user's market. It generates suggestions and synthesizes inputs; it never adjudicates. A "this product scores 7/10" output would set the user up to either over-trust the system or distrust it once it's wrong.

**Single-user MVP.** No teams, no sharing, no public marketplace. Security model is dead simple: every document lives under `users/{uid}/...`, and Firestore rules enforce ownership rigorously. Multi-user is V2 territory after the loop works for one person.

**Versioned hypotheses, not flat lists.** v2 of a hypothesis has a `parentHypothesisId` pointing to v1. The AI reads v1 learnings to propose what to change. The user's history of thinking is preserved as a tree.

**Lazy-init Anthropic client.** `lib/claude/client.ts` uses a `getAnthropic()` function rather than module-top instantiation, so the build does not require `ANTHROPIC_API_KEY` to be set. Runtime requests still fail fast with a clean error if the key is missing.

**Global budget cap, no per-user rate limit.** A single global monthly cap (`MONTHLY_BUDGET_USD = 100`) sits in `system/budget/months/{YYYY-MM}.spentUsd` as a runaway-loop guard. Per-user enforcement is deliberately off in MVP — logging is enough at this scale.

**No AI caching.** Each Anthropic call is fresh. Caching adds reliability bugs and removes the variability that the user actually wants (regenerate is supposed to be different).

## Roadmap

Done and pushed: E0 (skills extraction from AdTestLab) → E1.1 (project bootstrap) → E1.2 (shadcn UI primitives port) → E2.1 (auth) → E3 (data layer for products, hypotheses, viability, wizard sessions, AI usage) → E4 (pipeline dashboard + minimal Quick Add) → E6 (AI service layer) → E7.1 (wizard scaffolding + cards 1–3) → E7.2/7.3 (wizard cards 4–9 + Save & Finish).

Next:

- **E5** — full bulk Quick Add (spreadsheet table, multi-line paste, CSV import, OG-scraping via a 3rd-party service like Microlink)
- **E9** — Hypothesis Editor (full-page editor for a single hypothesis; lets the user enter actual KPIs, write learnings, change status to `tested`, and trigger AI v2 suggestion from learnings). This is the loop-closer.
- **E10** — Results & versioning (visual hypothesis tree, side-by-side compare table for ≤5 hypotheses)
- **E11** — Settings page (display name, AI usage stats, account deletion with cascade)
- **E12** — Polish and deploy

V2 (post-MVP):

- AdTestLab integration (auto-pull of actual results into a hypothesis via the `linkedAdTestLabId` field already in the schema)
- Chrome extension for one-click product capture from TikTok / AliExpress / Amazon
- Per-field AI assistance ("rewrite this hook" inline)
- Niche templates (preset avatars and angles for common verticals)
- Launch brief export (PDF or Notion)
- Cross-product hypothesis compare

V3 (further future): teams, public anonymized benchmarks, calibrated readiness scoring, creative generation, multi-language UI.

Explicitly out of scope, forever: AI-as-oracle features, spy-tool functionality, ad platform management.
